- **项目定位**：个人博客系统，支持文章发布、Markdown 写作、图片上传、分类管理、关于页配置和线上访问。
- **前端技术栈**：React 18、TypeScript、Vite、Tailwind CSS、React Router、Zustand、Axios。
- **后端技术栈**：Node.js、Express、SQLite、JWT、Multer。
- **部署架构**：Nginx 静态资源托管 + API 反向代理 + PM2 管理后端进程。
- **线上域名**：`manji.pro`，服务器 IP 为 `14.103.45.4`。
- **固定部署路径**：前端 `/blog/dist/`，后端 `/blog/api/`，PM2 配置 `/blog/`。

## v0.4 Markdown 写作能力

### 版本目标

优化写作体验，让后台文章编辑支持 Markdown、实时预览和图片插入。

### 主要调整

- 完成文章编辑页 `ArticleEdit`，支持新建和编辑文章。
- 引入 `marked`，建立 `src/utils/markdown.ts` Markdown 渲染工具。
- 支持 Markdown 实时预览，修复预览内容只显示前 500 字的问题。
- 支持点击上传图片，并自动把图片 Markdown 语法插入正文。
- 增加编辑器状态字段：标题、正文、分类、发布状态。
- 增加草稿自动保存机制，使用 `localStorage` 进行 2 秒防抖保存。

### 关键问题

- 早期预览逻辑存在截断，导致用户无法完整浏览正在编辑的文章。
- 编辑页面卡死或浏览器异常时，未保存内容容易丢失。

### 解决方案

- 移除编辑预览的固定截断，改为渲染完整内容。
- 增加本地草稿缓存，降低写作过程中的内容丢失风险。

### 阶段结果

后台写作体验从普通表单升级为可实时预览的 Markdown 写作工具。

## v0.6 上传与数据库修复

### 版本目标

解决实际写作和部署中暴露出的图片上传、数据库字段、服务稳定性问题。

### 主要调整

- 修复 Axios 请求拦截器对 `FormData` 错误设置 `Content-Type` 的问题。
- 修复上传接口中手动设置 `multipart/form-data` 导致 boundary 丢失的问题。
- 后端上传路由自动创建 `uploads` 目录，避免目录不存在导致上传失败。
- 在 Nginx 中增加 `client_max_body_size 10M;`，解决 1MB 以上图片上传失败。
- SQLite 启用 WAL 模式，提升并发读写稳定性。
- 增加 `busy_timeout` 和 `synchronous = NORMAL`，降低数据库锁冲突。
- 优化服务关闭和 PM2 重启时的数据库连接释放。

### 关键问题

- 1.8MB 图片上传失败，最终定位为 Nginx 默认上传体积限制。
- FormData 被错误设置请求头，导致服务端无法正确识别上传内容。
- SQLite 在开发和重启过程中可能出现锁等待问题。

### 阶段结果

图片上传链路和后端数据写入稳定性明显提升，系统更接近可长期运行状态。

## v0.7 摘要与封面调整

### 版本目标

根据实际写作习惯调整文章信息结构，减少封面图维护成本，增加摘要能力。

### 主要调整

- 移除封面图功能，不再在首页、列表页、详情页和后台管理中强依赖封面图。
- 增加文章摘要字段 `excerpt`。
- 在文章编辑页增加摘要输入框。
- 后端文章创建和更新接口支持 `excerpt`。
- 数据库 `articles` 表增加 `excerpt TEXT` 字段。
- 修复文章保存失败问题，补齐数据库迁移逻辑。
- 修复首页仍显示“暂无摘要”的问题，补齐 SQL 查询中的 `excerpt` 字段。

### 关键问题

- 前端已提交摘要，但数据库表结构没有对应字段，导致保存失败。
- 后端列表查询未返回 `excerpt`，导致首页无法显示已填写摘要。

### 阶段结果

文章卡片信息更完整，写作者不再需要维护封面图，发布流程更轻量。

## v0.8 文章排版优化

### 版本目标

提升文章阅读体验，重点优化图片尺寸、正文宽度、字体、行高和段落间距。

### 主要调整

- 对比参考 Jekyll、tw93 博客和 liruifengv.com 的文章排版方式。
- 明确图片宽度应跟随正文容器宽度，而不是无限放大。
- 设置 `.markdown-preview img` 为 `width: 100%; height: auto;`，让图片与正文行宽一致。
- 增加 `.images` 容器，用于两张图片并排展示。
- 优化 Markdown 段落、列表、引用、代码块、表格和图片的间距。
- 引入 Fraunces 作为标题字体，Nunito 作为正文字体。
- 通过 Google Fonts 在 `index.html` 中加载外部字体资源。
- 调整正文为 17px、行高 1.8，提升长文阅读舒适度。
- 避免给 H2 添加下划线分隔，保持更简洁的阅读风格。

### 关键问题

- 用户希望文章图片上传后自动保持合理宽度，而不是每张图都需要手动调整。
- HTML 自定义图片布局尝试存在转义和渲染问题，最终选择更稳妥的 Markdown + CSS 方案。
- 某次样式优化不符合预期，已按反馈回滚图片点击放大等不需要的效果。

### 阶段结果

文章页形成了更统一的阅读样式：图片贴合正文宽度、文字更松弛、段落更适合移动端和桌面端阅读。

## 开发过程复盘

### 做得好的地方

- 技术选型轻量，React + Express + SQLite 适合个人博客低成本部署。
- 功能迭代紧贴实际使用问题，很多改动来自真实写作、上传、部署和移动端访问反馈。
- 部署路径最终被固定，减少了前后端文件上传到错误目录的风险。
- Markdown、摘要、USER.md 和关于页让内容表达更自由。
- 移动端适配补齐后，博客更适合在微信和飞书里传播。

### 暴露的问题

- 早期没有 Git 版本记录，导致后期复盘只能按功能阶段整理，无法精确追踪每次变更。
- 数据库字段演进没有独立迁移系统，新增 `excerpt` 时出现过保存失败。
- 图片上传链路涉及前端请求头、后端目录、Nginx 限制，问题排查跨度较大。
- 部署流程前期没有脚本化，容易出现路径写错、文件漏传、服务未重启等问题。
- 样式优化有主观性，部分参考样式需要先预览再决定是否保留。

### 后续建议

- 从当前版本开始初始化 Git，并以 `v1.0.0` 作为第一个稳定版本。
- 每次需求完成后更新本文档，形成持续版本日志。
- 为数据库增加 `migrations/` 目录，所有表结构变化都通过脚本执行。
- 为图片上传增加前端大小提示和后端错误码，让失败原因更清晰。
- 部署前固定执行 `npm run build`、检查 `dist/assets`、检查 Nginx 配置、检查 PM2 状态。
- 移动端每次样式改动后，优先在微信和飞书内置浏览器中实测。

## v1.1.1 修复随想发布时间展示

### 版本目标

修复后台发布随想后，实际发布时间与前台/后台列表展示时间不一致的问题，确保新旧随想都按本地时区正确显示。

### 主要调整

- 后端创建随想时不再依赖 SQLite `CURRENT_TIMESTAMP` 默认值，改为显式写入 `created_at` 和 `updated_at` 的 ISO UTC 时间。
- 新增 `src/lib/date.ts`，统一处理 API 返回时间的解析和中文日期格式化。
- 兼容历史随想的 SQLite 时间格式 `YYYY-MM-DD HH:mm:ss`，将其按 UTC 解析后再转为本地时间展示。
- 首页「最近随想」、公开「随想」页和后台「随想管理」页统一使用新的日期格式化工具。

### 关键问题

- SQLite `CURRENT_TIMESTAMP` 生成的是 UTC 时间，但字符串没有时区标记。
- 浏览器 `new Date("YYYY-MM-DD HH:mm:ss")` 会把这种字符串当成本地时间解析，导致北京时间展示比真实发布时间慢 8 小时。
- 更新接口使用 `new Date().toISOString()`，创建和更新时间格式不一致，进一步放大了展示口径差异。

### 解决方案

- 创建随想时统一写入 ISO UTC 时间，后续新数据自带明确时区语义。
- 前端解析历史 SQLite 时间时自动补 `Z`，按 UTC 解析后用浏览器本地时区展示。
- 将日期格式化集中到一个工具文件，避免各页面重复写 `new Date(dateStr)`。

### 阶段结果

随想发布时间在首页、公开列表和后台管理中保持一致；历史数据无需迁移即可按正确时区展示。

## v1.1.2 修复文章列表排序口径

### 版本目标

修复文章列表排序与展示日期口径不一致的问题：列表按 `created_at` 排序、却展示 `published_at`，当二者不同时会出现「2025 年文章夹在两篇 2026 文章中间」的错位。

### 主要调整

- `api/routes/articles.ts` 公开列表与后台列表的 `ORDER BY` 由 `a.created_at DESC` 改为 `COALESCE(a.published_at, a.created_at) DESC`，与前端展示字段 `published_at || created_at` 对齐。
- 用 `COALESCE` 兜底：草稿（`published_at` 为 NULL）回退到 `created_at`，避免未发布内容排序异常。
- 修复 `deploy.local.sh` 后端同步遗漏：原脚本逐个 `scp` 仅同步部分后端文件（漏 `articles.ts`、`auth.ts`、`categories.ts`、`settings.ts`、`middleware/auth.ts`），导致本次排序修复首次部署后线上未生效；改为 `rsync` 整个 `api/` 目录（排除 `database/` 与 `uploads/`，绝不覆盖线上数据库），根治漏文件问题。

### 关键问题

- 前端 `ArticleList.tsx` 展示 `published_at || created_at`，后端排序却用 `created_at`，两个口径不统一。
- 当文章的 `created_at`（记录创建/迁移时间）与 `published_at`（对外发布日期）不同时，排序位置由 `created_at` 决定、显示日期由 `published_at` 决定，肉眼看到的就是错位。
- 历史数据中存在 `published_at` 格式不统一（ISO `...T...Z` 与纯日期 `YYYY-MM-DD` 混用），但均以 `YYYY-MM-DD` 开头，字符串排序前缀比较仍正确。

### 解决方案

- 排序字段与展示字段统一为「`published_at` 优先，缺失时回退 `created_at`」，根治口径分歧。
- 不改动历史数据：`published_at` 是博主自定义的对外发布日期，保留原值（如 id=10 展示 2025-11-17）即可，排序自然落在对应位置。

### 阶段结果

文章列表从左至右严格按 `published_at` 由近及远排列，展示日期与排序位置一致；草稿不受影响。
